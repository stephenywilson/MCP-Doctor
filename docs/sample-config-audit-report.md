# Sample MCP Config Audit Report

This is an example of the kind of report generated by:

```bash
mcp-doctor config-audit --config examples/risky-config-audit-config.json
```

## Summary

| Field | Value |
| --- | --- |
| Risk score | 100/100 |
| Risk level | Critical |
| Config files scanned | 1 |
| MCP servers detected | 3 |

## Findings by Severity

### CRITICAL

- Server command can execute arbitrary shell (`dangerous-shell`)
  - Remediation: Avoid launching MCP servers through a shell. Use a direct executable path and fixed argument list.
- Config exposes API keys or tokens in plaintext (`dangerous-shell`)
  - Remediation: Move secrets to environment variables or a secret manager.
- Filesystem access points to home directory or root directory (`broad-filesystem`)
  - Remediation: Restrict filesystem access to the smallest project directory needed by this server.

### HIGH

- Server uses package execution without clear version pinning (`broad-filesystem`, `github-unpinned`)
  - Remediation: Pin MCP server packages to reviewed versions.
- Server has network or external service access (`github-unpinned`)
  - Remediation: Use least-privilege API tokens and confirm the server only needs the listed network service.

### MEDIUM

- Missing server description
  - Remediation: Document the server purpose, owner, expected tools, and why each permission is needed.
- No allowlist policy found
  - Remediation: Use allowlists where supported by the MCP client or server wrapper.

## Safe Configuration Recommendations

- Restrict filesystem paths to specific project directories.
- Move secrets to environment variables and rotate any plaintext tokens found in config files.
- Pin package-based MCP servers to reviewed versions.
- Use least-privilege credentials for network-connected MCP servers.
- Use allowlists where possible for tools, paths, and server permissions.

MCP Doctor runs locally and does not send your MCP configuration, API keys, or project files to any external service.
